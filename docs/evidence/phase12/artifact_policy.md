# Phase 12 Artifact Evidence Policy

## Purpose

This document defines how GitHub Actions artifacts are used as lightweight evidence in Phase 12.

The goal is to make CI(Continuous Integration) results traceable without pretending that GitHub Actions artifacts are permanent audit storage.

## Evidence model

| Evidence layer | Role | Location |
|---|---|---|
| Temporary evidence | Raw CI result files generated by each run | GitHub Actions artifact |
| Long-term index | Human-readable evidence index that records commit, tag, run, artifact name, and result | `docs/evidence/phase12/phase12_ci_evidence_index.md` |

## Artifact naming rule

Use unique artifact names that include the workflow role, `github.run_id`, and `github.run_attempt`.

Suggested names:

```text
phase12-python-test-evidence-${{ github.run_id }}-${{ github.run_attempt }}
phase12-cpp-sils-ci-evidence-${{ github.run_id }}-${{ github.run_attempt }}
phase12-python-cpp-comparison-evidence-${{ github.run_id }}-${{ github.run_attempt }}
```

## Artifact directory structure

Recommended internal artifact structure:

```text
reports/
  ci/
    metadata.json
    tool-versions.txt
  python/
    pytest-results.xml
    python-scenario-results.json
    python-scenario-summary.md
  cpp/
    ctest-results.xml
    cpp-scenario-results.json
    cpp-scenario-summary.md
  compare/
    python-cpp-comparison.json
    python-cpp-comparison-summary.md
```

Not all files need to exist from the beginning. However, missing evidence files should be intentional and documented.

## Metadata minimum set

Each workflow should generate a metadata file such as `reports/ci/metadata.json`.

Example:

```json
{
  "phase": "Phase 12",
  "workflow": "<workflow name>",
  "commit_sha": "<github.sha>",
  "run_id": "<github.run_id>",
  "run_attempt": "<github.run_attempt>",
  "scenario_suite": "SC_01_to_SC_05",
  "generated_at_utc": "<timestamp>",
  "result": "passed_or_failed"
}
```

## Upload behavior

Artifact upload should fail when expected files are missing.

Recommended setting:

```yaml
if-no-files-found: error
```

This prevents a workflow from appearing successful while evidence files are missing.

## Retention policy

GitHub Actions artifact and log retention is time-limited. Do not treat artifacts as permanent SQA(Software Quality Assurance) evidence.

Long-term evidence should be a small index committed to the repository under `docs/evidence/phase12/`.

## What not to store

Do not store any of the following in artifacts:

- Company information
- Customer information
- Real project data
- Real vehicle data
- Internal documents
- Customer-specific requirements
- Customer-specific logs
- Credentials or secrets

All scenarios, requirements, test data, and expected results must remain fictional and general-purpose for this personal PoC(Proof of Concept).
